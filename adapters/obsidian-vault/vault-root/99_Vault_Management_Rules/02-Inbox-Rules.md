# Inbox 规则

`3-Inbox` 是原始材料层，目标是低摩擦收集，而不是精细整理。

## 目录职责

`3-Inbox` 保存尚未消化的材料，包括：

- 网页剪藏
- 书籍导出
- 论文导出
- 外部教程
- 临时调研资料
- 尚未判断价值的 AI 输出

## 子目录建议

```text
3-Inbox/
  00-Guide/
  Clippings/
  Ideas/
  Y-Books2MD/
  Y-Paper2MD/
  AI-Chats/
  Temp/
```

`AI-Chats/`、`Ideas/` 和 `Temp/` 可按需要创建，不必预先建空目录。

## 原始性原则

`3-Inbox` 中的内容默认视为原始材料。可以修复格式、补全图片本地化，但不应直接改写成自己的观点。

## 什么时候离开 Inbox

- 提炼出长期判断 -> `4-Wiki`
- 进入具体项目推进 -> `5-Projects`
- 变成正式文档材料 -> `6-Outputs`
- 不再需要但不删除 -> `7-Archive`

## 剪藏处理原则

技术文档类剪藏可删除装饰性符号、平台化结尾、无关推广和站内搜索外链；保留核心信息、表格、步骤、数据、来源链接。


## 剪藏标题规则

处理剪藏内容时，AI 必须先判断标题是否合适：

- 标题是否是一句直观的话。
- 看标题是否能知道这篇内容主要讲什么。
- 看标题是否能知道核心结论、方法或判断。
- 是否存在空泛标题，如“资料”“笔记”“总结”“相关”“一个方法”。
- 是否只是平台原题、营销标题、悬念标题、情绪标题，但不能概括正文内容。

如果标题不直观，AI 应先阅读正文，提炼核心结论，再把标题改成“看标题就知道讲了什么结论”的标题。

推荐标题：

```text
文件夹只负责放置，链接负责发现.md
Atlas 适合做知识地图而不是正文仓库.md
论文剪藏应先保留来源再提炼观点.md
```

不推荐标题：

```text
知识库整理.md
一个超好用的方法.md
资料总结.md
关于 Obsidian.md
```

标题修改边界：

- 单篇剪藏标题可在整理时直接修正。
- 需要重命名 3 个以上文件时，必须先列出“原文件名 -> 建议文件名 -> 修改理由”，等待用户确认。
- 修改标题时应同步检查正文一级标题和 frontmatter 的 `title` 字段是否需要一致更新。
- 不为了标题好看而改变原文事实、夸大结论或编造正文没有的判断。
## 图片处理

外部图片下载到 `1-Attachment/`，正文替换为本地 wiki-link：`![[image-name.jpg]]`。图片标题如有必要，用普通斜体放在图片下方。

## 禁止事项

- 不把 `3-Inbox` 当长期知识库。
- 不为了美观过度整理原始材料。
- 不删除原始材料中的关键数据、实验条件、引用信息。
- 不把未读完材料迁入 `4-Wiki`。
